Auditing Dynamics 365 & Dataverse for Data Exposure

Summary: A security assessment of a customer's Dynamics 365 environment. The brief was simple: find where the CRM's crown-jewel data — contacts, cases, and the PII attached to them — could leak. What surfaced was a familiar story. A Power Pages site served Dataverse tables over anonymous OData, several security roles handed out organization-wide read on sensitive tables, PII columns had no field-level security, and an application user was sitting on System Administrator. None of it was a "hack" — it was configuration, and configuration is exactly what an attacker enumerates first.

Why Dynamics 365 is a juicy target

Dynamics 365 sits on top of Dataverse (the platform formerly known as the Common Data Service / CDS). Every "app" — Sales, Customer Service, Field Service, a bespoke model-driven app — is really just a set of views over Dataverse tables. Those tables hold the things attackers and regulators care about most: contact (names, emails, phone numbers, addresses), incident (support cases, often full of customer detail), account, and whatever custom tables the business bolted on to hold contracts, claims, or worse — national IDs and payment references.

The security model is genuinely capable: role-based access, record ownership, business units, field-level security, and hierarchical sharing. The problem is that every one of those controls is opt-in and easy to loosen, and there are several front doors — the classic web client, the Dataverse Web API, Power Pages/Portals, Power Automate, and application users. An audit has to walk each door and rattle the handle.

Test 1 — Is the environment fenced off at all?

The first and cheapest question: does the environment have a security group? In the Power Platform admin center, an environment with no security group assigned is reachable by every licensed user in the tenant — not just the team that owns the app. On a large tenant that can mean thousands of accounts, most of whom have no business being anywhere near the CRM data.

Here it was unset. Combined with a couple of broad security roles (next section), that meant the blast radius of any single phished employee was the entire customer database.

Test 2 — Power Pages anonymous OData (the big one)

This is the finding that keeps showing up across the industry, and it was present here too. Power Pages exposes Dataverse through a portal Web API. Access is gated by three things that all have to line up: site settings that enable the Web API for a table, a table permission, and a web role the permission is granted to. The trap is the built-in Anonymous Users web role — grant a table permission to it (or to the "Authenticated Users" role while self-registration is open) and you have just published Dataverse to the internet.

Enumeration starts with the metadata document, which happily lists every entity set the API knows about — no auth required if the Web API is on:

GET https://portal.example.com/_api/$metadata
# → EntitySet Name="contacts" ... "incidents" ... "accounts" ...

From there it is a single OData query to pull records. Standard OData operators ($select, $filter, $top, $orderby) all work, so an attacker doesn't scrape a UI — they page through the table:

GET https://portal.example.com/_api/contacts?$select=fullname,emailaddress1,telephone1,address1_city&$top=5000

{
  "value": [
    { "fullname": "Jane R.",  "emailaddress1": "jane.r@example.com",  "telephone1": "+1-555-0142", "address1_city": "Austin" },
    { "fullname": "Omar K.",  "emailaddress1": "omar.k@example.com",  "telephone1": "+1-555-0198", "address1_city": "Denver" },
    ... 4,998 more ...
  ]
}

The portal's own JavaScript needs a request-verification token for writes, but reads over the Web API returned data to an unauthenticated client. The fix is not clever: the table permissions granted Global scope read to a role that anonymous users inherited. Scope should almost never be Global on a portal — it should be Contact, Account, or Self so a signed-in user sees only their own related records.

Test 3 — Over-broad security roles & the Web API

Away from the portal, the authenticated Dataverse Web API is only as safe as the security roles behind it. I exported the roles and looked specifically at the read privilege depth on sensitive tables. Dataverse depths run User → Business Unit → Parent-Child BU → Organization. A green-filled "Organization" circle on contact read means every holder of that role can read every contact in the company.

Any user with such a role can dump the table with the same OData a portal uses, just authenticated:

GET https://org.crm.dynamics.com/api/data/v9.2/contacts?$select=fullname,emailaddress1,telephone1&$top=5000
Authorization: Bearer <user-or-app-token>

Two custom roles cloned from Salesperson had quietly been bumped to Organization-level read across contact, incident, and a custom claim table. Nobody remembered doing it — it had been copied forward across environments for years. That is the normal way privilege sprawl happens: nobody grants "read everything," they clone a role and nudge one circle.

Test 4 — Field-level security on the PII that matters

Role privileges are per-table; field-level security is how you protect individual columns like a national ID or a bank reference regardless of who can see the row. You mark a column as secured, then a Field Security Profile decides who can Read / Update / Create that column.

The custom claim table had a national_id column holding exactly the kind of data you never want to explain to a regulator — and it was not secured. That meant it flowed through both the authenticated Web API and (because of Test 2) the portal feed, in cleartext:

GET .../claims?$select=claim_number,national_id,claimant_email
# national_id returned in full — no field security profile in the way

Securing the column and scoping a profile to just the claims-handling team removed it from every response for everyone else, without changing a single security role.

Test 5 — Application users & service principals

Application users (service principals registered in Dataverse) are the accounts nobody watches. They authenticate with a client secret or certificate and carry whatever security role they were given. I enumerated them and checked their roles:

GET https://org.crm.dynamics.com/api/data/v9.2/systemusers?$select=fullname,applicationid,isdisabled&$filter=applicationid ne null

One integration app — wired into a Power Automate flow — held System Administrator. Its client secret was pasted into the flow's connection and had no expiry. Anyone who could read that flow definition, or who lifted the secret from source control, would inherit god-mode over the whole environment. Application users should get a narrow custom role scoped to the exact tables they integrate, secrets should be short-lived and stored in Key Vault, and certificate auth is preferable to a shared secret.

Remediation & hardening checklist

AreaAction
Portal / Power PagesReview every table permission; remove Global scope; never grant read to the Anonymous role unless the data is truly public. Turn off the Web API for tables that don't need it.
Environment fencingAssign a security group to every non-trivial environment so only intended users can access it.
Security rolesLeast privilege: keep read depth at User/BU for sensitive tables. Audit cloned roles for drift; avoid Organization scope on PII tables.
Field-level securitySecure PII/financial columns (national IDs, payment refs) and scope Field Security Profiles to the teams that genuinely need them.
Application usersNarrow custom roles per integration; short-lived secrets in Key Vault or certificate auth; disable unused app users.
MonitoringEnable Dataverse auditing on sensitive tables; surface bulk RetrieveMultiple and export activity via Purview / the audit log; alert on anonymous portal volume spikes.

The takeaway

Not one of these findings needed an exploit. Dynamics 365 gave the attacker everything through the front door — a well-documented OData API and a security model that had been loosened one reasonable-looking click at a time. Auditing it isn't about finding a bug; it's about walking every door the platform opens and asking who is actually standing on the other side. Do that on a schedule, because the config drifts back.
