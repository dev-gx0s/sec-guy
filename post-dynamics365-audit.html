Auditing Dynamics 365 & Dataverse for Data Exposure

Summary: A security assessment of a Dynamics 365 environment, driven entirely through the Dataverse Web API. Given one ordinary authenticated token, how much of the organization can it map? Everything below runs against a single endpoint, https://<org>.crm.dynamics.com/api/data/v9.2/, and follows the reconnaissance an attacker runs the moment they land: WhoAmI to fix the identity, business units to learn the org's shape, user recon to build a directory, and admin recon to find the accounts worth stealing. No exploit — just OData and a security model that had drifted.

Why Dynamics 365 is a juicy target

Dynamics 365 sits on top of Dataverse (the platform formerly known as the Common Data Service). Every "app" — Sales, Customer Service, a bespoke model-driven app — is really views over Dataverse tables, and the platform's own plumbing is stored in tables too: systemuser, businessunit, role. That means once you hold a token, the OData v4 Web API under /api/data/v9.2/ lets you read not just the business data but the entire identity and privilege model — who exists, how the org is carved up, and who holds the keys.

The security model is capable — role-based access, ownership, business units, field-level security — but every control is opt-in and easy to loosen. Reconnaissance is how an attacker finds the loose one. This engagement walks that recon exactly as an adversary would, then reports what it exposed.

The one endpoint that matters

The whole assessment runs against a single host and its Web API. These are the tables and functions the recon touches:

PurposeEndpointReveals
Identity/api/data/v9.2/WhoAmIUserId, BusinessUnitId, OrganizationId
Org structure/api/data/v9.2/businessunitsBusiness-unit hierarchy & ownership boundaries
Directory/api/data/v9.2/systemusersEvery user & application user, emails, status
Privilege/api/data/v9.2/roles + role associationsWho holds which security role

Authentication

Everything that follows needs a bearer token for the Dataverse org. The Web API is protected by Entra ID (Azure AD), and the quickest way to get an interactive token as yourself is the Azure CLI — az login opens a browser sign-in against the tenant:

# Opens a browser to sign in to Entra ID (Azure AD)
az login

# Sanity check: can this identity mint a token for the Dataverse org?
az account get-access-token --resource https://org.crm.dynamics.com

From there, a few lines of Python reuse that CLI session, request a token scoped to the org (<org>/.default), and set up a requests session with the OData headers the Web API expects. Every call in the rest of this post rides on this session:

Python
import requests
from azure.identity import AzureCliCredential   # pip install azure-identity requests

ORG = "https://org.crm.dynamics.com"

# Reuse the `az login` session and get a token scoped to the Dataverse org
token = AzureCliCredential().get_token(f"{ORG}/.default").token

session = requests.Session()
session.headers.update({
    "Authorization":    f"Bearer {token}",
    "Accept":           "application/json",
    "OData-MaxVersion": "4.0",
    "OData-Version":    "4.0",
})

# Smoke test — the very first recon call
r = session.get(f"{ORG}/api/data/v9.2/WhoAmI")
print(r.status_code, r.json())

For an unattended assessment you'd swap AzureCliCredential for ClientSecretCredential (an app registration's client ID + secret) or CertificateCredential — the app just needs to be provisioned as an application user in Dataverse. With a live session in hand, the recon begins.

WhoAmI

Every Dataverse assessment starts here. Before enumerating anything you need to know which identity your token represents, and which organization and business unit it lands in. The WhoAmI function answers all three in a single GET (it still needs a valid bearer token — it just takes no parameters):

GET https://org.crm.dynamics.com/api/data/v9.2/WhoAmI
Authorization: Bearer <token>

{
  "@odata.context": ".../$metadata#Microsoft.Dynamics.CRM.WhoAmIResponse",
  "UserId":         "a1b2c3d4-5e6f-4a7b-8c9d-0e1f2a3b4c5d",
  "BusinessUnitId": "6f4d0a11-77aa-4b2c-9d3e-1122334455aa",
  "OrganizationId": "9e8f7d6c-3b2a-4c1d-8e9f-aabbccddeeff"
}

Two GUIDs drive everything that follows. UserId is the primary key of my own row in systemuser (user recon). BusinessUnitId is my row in businessunit — the anchor for mapping the org.

Business Units

Business units are the ownership and scoping boundaries of Dataverse. A security role's read depth of "Business Unit" means this BU and below — so before you can reason about what any role actually reaches, you have to map the tree. The businessunit table hands it over, parent pointers and all:

GET https://org.crm.dynamics.com/api/data/v9.2/businessunits?$select=name,isdisabled&$expand=parentbusinessunitid($select=name)
Authorization: Bearer <token>

{
  "value": [
    { "name": "Contoso (root)",     "parentbusinessunitid": null },
    { "name": "EMEA Sales",         "parentbusinessunitid": { "name": "Contoso (root)" } },
    { "name": "EMEA Support",       "parentbusinessunitid": { "name": "Contoso (root)" } },
    { "name": "Claims Processing",  "parentbusinessunitid": { "name": "EMEA Support" } }
  ]
}

The same call in Python, walking the tree off the shared session:

Python
r = session.get(f"{ORG}/api/data/v9.2/businessunits", params={
    "$select": "name,isdisabled",
    "$expand": "parentbusinessunitid($select=name)",
})
for bu in r.json()["value"]:
    parent = (bu.get("parentbusinessunitid") or {}).get("name", "—")
    print(f"{bu['name']:<20} parent={parent}")

Cross-referencing my BusinessUnitId from WhoAmI against this tree tells me exactly where my identity sits and, therefore, what a "Business Unit"-scoped role would and wouldn't cover. It also names the sensitive units — Claims Processing was the one worth remembering.

User Reconnaissance

The systemuser table is the org directory, and any role with even modest read on it turns into a bulk export of names, login names, email addresses, and even postal addresses — a ready-made phishing, password-spray, and physical-targeting list. I paged the enabled users, tagged by business unit:

GET https://org.crm.dynamics.com/api/data/v9.2/systemusers
      ?$select=fullname,domainname,internalemailaddress,address1_composite,isdisabled
      &$expand=businessunitid($select=name)
      &$filter=isdisabled eq false&$top=5000
Authorization: Bearer <token>

{
  "value": [
    { "fullname": "Jane R.", "domainname": "jane.r@example.com", "address1_composite": "220 King St W, Toronto, ON M5H 1K4, Canada", "businessunitid": { "name": "EMEA Sales" } },
    { "fullname": "Omar K.", "domainname": "omar.k@example.com", "address1_composite": "14 Rue de la Paix, 75002 Paris, France",     "businessunitid": { "name": "Claims Processing" } },
    ... hundreds more ...
  ]
}

In Python, that's a single paged request straight into a directory dump:

Python
r = session.get(f"{ORG}/api/data/v9.2/systemusers", params={
    "$select": "fullname,domainname,internalemailaddress,address1_composite,isdisabled",
    "$expand": "businessunitid($select=name)",
    "$filter": "isdisabled eq false",
    "$top":    5000,
})
for u in r.json()["value"]:
    bu = (u.get("businessunitid") or {}).get("name", "")
    print(u["fullname"], u["domainname"], u.get("address1_composite"), "|", bu)

Then I turned the lens on my own token — a self-lookup by the UserId from WhoAmI, expanding the role association to see exactly what I hold:

GET https://org.crm.dynamics.com/api/data/v9.2/systemusers(a1b2c3d4-5e6f-4a7b-8c9d-0e1f2a3b4c5d)
      ?$select=fullname,applicationid
      &$expand=systemuserroles_association($select=name)

{
  "fullname":      "Service Account – Integrations",
  "applicationid": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
  "systemuserroles_association": [
    { "name": "Salesperson (custom)" },
    { "name": "System Administrator" }
  ]
}
Python
me = session.get(f"{ORG}/api/data/v9.2/WhoAmI").json()

r = session.get(f"{ORG}/api/data/v9.2/systemusers({me['UserId']})", params={
    "$select": "fullname,applicationid",
    "$expand": "systemuserroles_association($select=name)",
})
u = r.json()
roles = [x["name"] for x in u["systemuserroles_association"]]
print(u["fullname"], "| app user:", bool(u.get("applicationid")), "| roles:", roles)

That last line is the tell: my token isn't a person at all — it's an application user (note the applicationid) carrying System Administrator. User recon just became admin recon.

Admin Reconnaissance

The final move is to enumerate privilege directly: who holds the powerful roles, and which of those accounts are non-human. OData lets you filter systemusers by role association, so one request lists every System Administrator in the org:

GET https://org.crm.dynamics.com/api/data/v9.2/systemusers
      ?$select=fullname,domainname,applicationid,isdisabled
      &$filter=systemuserroles_association/any(r:r/name eq 'System Administrator')
Authorization: Bearer <token>

And the accounts nobody watches — application users (service principals) — fall out of a single filter on applicationid:

GET https://org.crm.dynamics.com/api/data/v9.2/systemusers?$select=fullname,applicationid&$filter=applicationid ne null
Python
# Every System Administrator — flag the non-human ones
r = session.get(f"{ORG}/api/data/v9.2/systemusers", params={
    "$select": "fullname,domainname,applicationid,isdisabled",
    "$filter": "systemuserroles_association/any(r:r/name eq 'System Administrator')",
})
for u in r.json()["value"]:
    kind = "APP" if u.get("applicationid") else "user"
    print(f"[{kind}] {u['fullname']} — {u.get('domainname')}")

# Application users (service principals) — the accounts nobody watches
apps = session.get(f"{ORG}/api/data/v9.2/systemusers", params={
    "$select": "fullname,applicationid",
    "$filter": "applicationid ne null",
}).json()["value"]
print(f"{len(apps)} application users found")

The results were the finding. Alongside a healthy handful of interactive admins, the integration app from the user-recon step held System Administrator — and its client secret was pasted straight into a Power Automate flow connection with no expiry. Anyone who could read that flow definition, or who lifted the secret from source control, would inherit god-mode over the entire environment. Application users should carry a narrow custom role scoped to the exact tables they integrate; secrets should be short-lived, stored in Key Vault, and certificate auth preferred over a shared secret.

Custom Entity Discovery

Identity and privilege are only half the picture — the other half is where the corporate data actually lives. The out-of-box tables (contact, account, incident) are predictable; the sensitive material tends to sit in custom tables a maker built for the business — contracts, claims, payroll — which carry a publisher prefix like cr123_, contoso_, or new_. The metadata service hands the whole list over, filtered to just the custom entities so you skip the hundreds of system tables:

GET https://org.crm.dynamics.com/api/data/v9.2/EntityDefinitions
      ?$select=LogicalName,EntitySetName,DisplayName
      &$filter=IsCustomEntity eq true
Authorization: Bearer <token>

{
  "value": [
    { "LogicalName": "cr123_claim",     "EntitySetName": "cr123_claims",     "DisplayName": { ... "Claim" } },
    { "LogicalName": "cr123_contract",  "EntitySetName": "cr123_contracts",  "DisplayName": { ... "Contract" } },
    { "LogicalName": "contoso_payroll", "EntitySetName": "contoso_payrolls", "DisplayName": { ... "Payroll" } }
  ]
}

The names alone map the crown jewels. Ranking the discovered custom tables by how much they'd hurt in a breach gives a clear list to work down:

Entity set (OData)Logical nameLikely sensitive contentsPriority
cr123_claimscr123_claimClaim records — claimant name, national ID, email, payout amountsHigh
contoso_payrollscontoso_payrollSalaries, bank account & sort codes, tax identifiersHigh
cr123_contractscr123_contractCounterparties, contract values, signed termsMedium
new_onboardingsnew_onboardingNew-hire ID scans, home addresses, emergency contactsMedium

Each EntitySetName is a live OData collection, so reading one is the same call as any built-in table — and because these tables are often stood up quickly, they rarely have field-level security on the columns that need it most:

GET https://org.crm.dynamics.com/api/data/v9.2/cr123_claims?$select=cr123_claimnumber,cr123_nationalid,cr123_claimantemail&$top=50
Authorization: Bearer <token>
Python
# 1) List every custom table
defs = session.get(f"{ORG}/api/data/v9.2/EntityDefinitions", params={
    "$select": "LogicalName,EntitySetName",
    "$filter": "IsCustomEntity eq true",
}).json()["value"]
for e in defs:
    print(e["LogicalName"], "->", e["EntitySetName"])

# 2) Read one straight off its EntitySetName
r = session.get(f"{ORG}/api/data/v9.2/cr123_claims", params={
    "$select": "cr123_claimnumber,cr123_nationalid,cr123_claimantemail",
    "$top":    50,
})
for row in r.json()["value"]:
    print(row.get("cr123_claimnumber"), row.get("cr123_nationalid"))

Here the cr123_claim table held a cr123_nationalid column in cleartext, with no field security profile in the way — exactly the kind of data you never want to explain to a regulator, one OData query from any authenticated token.

Annotations (Note Attachments)

Columns are only half the data. Users attach files to records as Notes, and every one of those files lives in the annotation table with the bytes inline as base64 in documentbody. Filtering on isdocument eq true lists every attachment in the org — contracts, ID scans, spreadsheets — with the record type it hangs off:

GET https://org.crm.dynamics.com/api/data/v9.2/annotations
      ?$select=filename,mimetype,subject,objecttypecode
      &$filter=isdocument eq true
Authorization: Bearer <token>

{
  "value": [
    { "filename": "passport_scan.pdf",    "mimetype": "application/pdf", "objecttypecode": "contact" },
    { "filename": "signed_contract.docx", "mimetype": "application/...", "objecttypecode": "cr123_contract" },
    ... more ...
  ]
}

Add documentbody to the select on a single row and you get the file itself. In Python that's list-then-download — every attached document walked out over the Web API:

Python
import base64, pathlib

notes = session.get(f"{ORG}/api/data/v9.2/annotations", params={
    "$select": "annotationid,filename,mimetype",
    "$filter": "isdocument eq true",
    "$top":    5000,
}).json()["value"]

for n in notes:
    blob = session.get(f"{ORG}/api/data/v9.2/annotations({n['annotationid']})",
                       params={"$select": "documentbody"}).json()["documentbody"]
    pathlib.Path(n["filename"]).write_bytes(base64.b64decode(blob))
    print("saved", n["filename"], n["mimetype"])

Activity Attachments

Files attached to activities land in the activityfileattachment table — the modern File-column store. Unlike Notes, the bytes aren't inline base64; each row points at a fileattachment record, and you pull the content from that record's file-download endpoint. So it's a two-hop grab: enumerate the links, then stream each file:

GET https://org.crm.dynamics.com/api/data/v9.2/activityfileattachments
      ?$select=activityfileattachmentid,_objectid_value,_fileattachmentid_value
Authorization: Bearer <token>
Python
import pathlib

links = session.get(f"{ORG}/api/data/v9.2/activityfileattachments", params={
    "$select": "activityfileattachmentid,_fileattachmentid_value",
    "$top":    5000,
}).json()["value"]

for a in links:
    fid  = a["_fileattachmentid_value"]
    meta = session.get(f"{ORG}/api/data/v9.2/fileattachments({fid})",
                       params={"$select": "filename,mimetype"}).json()
    data = session.get(f"{ORG}/api/data/v9.2/fileattachments({fid})/file/$value").content
    pathlib.Path(meta["filename"]).write_bytes(data)
    print("saved", meta["filename"], meta["mimetype"])

Workflows & Processes

The workflow table stores every automation in the org — classic workflows, business process flows, actions, and modern cloud flows (Power Automate) — distinguished by category. Their definitions (clientdata JSON for modern flows, xaml for classic) map the plumbing: which systems the org integrates with, what triggers on which records, and — more often than anyone would like — hard-coded URLs, connection references, and the occasional secret:

GET https://org.crm.dynamics.com/api/data/v9.2/workflows
      ?$select=name,category,statecode,createdon
      &$filter=statecode eq 1
Authorization: Bearer <token>
Python
CAT = {0: "Workflow", 1: "Dialog", 2: "Business Rule",
       3: "Action", 4: "Business Process Flow", 5: "Modern Flow"}

flows = session.get(f"{ORG}/api/data/v9.2/workflows", params={
    "$select": "name,category,statecode,clientdata,xaml",
}).json()["value"]

for f in flows:
    print(CAT.get(f["category"], f["category"]), "-", f["name"])
    # clientdata / xaml frequently leak endpoints, connection refs, and secrets
    defn = f.get("clientdata") or f.get("xaml") or ""
    if "https://" in defn:
        print("   contains hard-coded URL(s)")

The takeaway

Not one step here needed an exploit. From a single token, four OData calls — WhoAmI, business units, users, admins — reconstructed the org's entire identity and privilege map and surfaced a service principal with standing god-mode. Dynamics 365 gives that away through the front door to anyone with a role, because the security model had been loosened one reasonable-looking click at a time. Audit it on a schedule and walk the recon yourself — because the config drifts back.
